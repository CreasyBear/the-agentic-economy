An autonomous agent can pay for an API call. It can open a session. It can stream micropayments at $0.00001 per request. It can settle in stablecoin, in fiat, in Lightning. What it cannot do is bear legal responsibility for any of it.

In March 2026, three major payment protocols launched within fourteen days. Stripe released the Machine Payments Protocol on March 18, backed by a $500 million Series A at a $5 billion valuation. MoonPay launched the Open Wallet Standard on March 23, with fifteen contributors including PayPal, Circle, and the Ethereum Foundation. Coinbase reports x402 has crossed 140 million cumulative transactions with over $600 million in payment volume.

The market response was confusion. Builders asked: should I integrate x402 or MPP? Is OWS competing with ERC-8004? Does Google's AP2 replace both?

The question itself reveals the problem. The protocols are not competitors. They are layers.

> The agent payments stack was built in parallel. Nobody drew the map.

This is not a protocol comparison. It is a stack analysis. Five layers. Sixteen protocols. One economy emerging beneath the coordination cost. The specifications exist. Production integration is early.

The economy runs on identity. Contracts require parties. Transactions require authorization. Liability requires attribution. Every payment system assumes an identity at the point of decision.

Agents do not have identity in the systems that were built for humans.

ERC-8004 established on-chain registries for agents. 130,000 agents registered across twenty EVM chains. The registry includes identity, reputation, and validation components. An agent can prove it exists on-chain. It can accumulate reputation. It can be validated by third parties.

AIP launched on March 26 with a different approach. JWT tokens signed with EdDSA. Authorization, payment, and trust in a single envelope. Delegation chains allow an agent to prove it was authorized to act on behalf of a principal. Fiat-first design — built for the payment rails that exist, not the ones that might exist.

Skyfire's Know Your Agent protocol uses JWT identity with F5 partnership for enterprise deployment. The question is not whether agents can be identified. The question is whether identification creates liability.

Consider the situation plainly. An agent pays for an API call via x402. The payment is authorized by the agent's wallet. The wallet is linked to an ERC-8004 registry entry. The registry proves the agent exists. It does not prove the agent was authorized to spend. It does not create legal existence. It does not enable the agent to be sued.

The identity layer solves discovery. It does not solve accountability.

Before March 23, every payment protocol required its own wallet infrastructure. x402 required a wallet that could sign HTTP payment requests. MPP required a wallet that could manage session keys. ERC-8004 required a wallet that could update registry entries. An agent running three protocols needed three wallets, three key management systems, three authorization flows.

MoonPay's Open Wallet Standard changed the layer beneath the protocols.

OWS is not a payment protocol. It is a wallet standard. MIT-licensed. Available on npm, PyPI, and GitHub. Eight chain families supported. Fifteen contributors including PayPal, Circle, and both the Ethereum and Solana Foundations.

The standard provides an encrypted vault. A policy engine. Protocol-agnostic signing. When x402 returns a payment request, OWS produces the signed authorization. When MPP opens a session and streams micropayments, OWS signs each payment within the agent's authorized limits. When an agent needs to update its ERC-8004 registry entry, OWS signs the transaction.

Ivan Soto-Wright, CEO of MoonPay, stated the gap plainly: "The agent economy has payment rails. It didn't have a wallet standard. We built one, open-sourced it, and now the full stack exists."

The strategic insight: OWS does not compete with x402 or MPP. It makes them more valuable. Any protocol that requires a signed transaction now has a common wallet to call. Any tool or agent that supports OWS can access any payment protocol without changing wallet logic.

This is the layer that was missing. Production deployments are early. The specification is complete.

Agents need to find services before they can pay for them. Model Context Protocol enables agents to access tools through a standardized interface. Google's Agent-to-Agent protocol connects fifty partners for agent-driven commerce. AP2 — the Agent Payments Protocol — sits alongside A2A as the payment authorization layer.

The communication layer is where payment intent is expressed. An agent discovers a service through MCP or A2A. It negotiates terms. It receives a payment request. The payment request is then handled by the wallet layer and settled by the payment layer.

Google's approach separates authorization from settlement. AP2 uses Verifiable Digital Credentials to prove the agent was authorized to pay. The actual money movement happens through x402, MPP, or another settlement protocol. AP2 does not move money. It proves the agent had permission to move money.

This separation matters. Authorization without settlement is a signature. Settlement without authorization is theft. The communication layer binds them.

Two paths are possible. The first: AP2 becomes the authorization standard, decoupled from settlement. The second: payment protocols integrate authorization, and AP2 remains Google-specific. The infrastructure is neutral. The market will decide.

The payment layer is where value actually moves. Two protocols dominate. One handles authorization only.

x402 revived the HTTP 402 "Payment Required" status code. Created by Coinbase and Cloudflare in September 2025. Coinbase reports 140 million transactions and $600 million in payment volume. Pay-per-request at $0.001 per call. No account required. No API keys. Authenticate with a wallet, pay with stablecoin, access the service.

The design of x402 is minimal. It does not attempt to solve all problems. It solves one problem: how does an agent pay for an HTTP request without human intervention? The answer: embed the payment in the HTTP flow itself.

MPP launched on March 18 with a different architecture. Also uses HTTP 402. Also enables agent payments. But MPP adds a session layer. An agent authorizes a spending limit upfront — say, $10 for the next hour. The agent then makes hundreds or thousands of requests within that session. Each request costs $0.00001. Settlement is batched off-chain, finalized on-chain.

The session primitive changes the economics. x402 at $0.001 per request is viable for low-volume agent traffic. MPP at $0.00001 per request — ten cents per million requests — enables high-frequency agent workflows. An agent can query an API every second for an hour and spend less than a dollar.

Stripe's integration advantage is not the protocol. It is the existing infrastructure. MPP is built into Stripe's PaymentIntents API. Businesses on Stripe can accept agent payments in a few lines of code. Funds settle into the business's existing balance, in their default currency, on their standard payout schedule. The same infrastructure used for human payments handles agent payments.

Visa extended MPP to card payments. Lightspark extended it to Lightning. MPP supports stablecoins natively on Tempo's Layer 1 blockchain. The protocol is payment-method-agnostic by design.

The market is not choosing between x402 and MPP. It is using both. x402 for permissionless, crypto-native settlement. MPP for multi-rail, high-volume sessions. AP2 for enterprise authorization. Stripe and Coinbase are competing for stack ownership, not collaborating on layer definition.

> The infrastructure for agents to pay is neutral with respect to which protocol they use. The market will use all of them.

The principal bears the liability. The agent bears nothing. Two paths are possible. The first: liability frameworks emerge. Identity, insurance, legal standing for agents.
